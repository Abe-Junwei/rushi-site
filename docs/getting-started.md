# Rushi 使用说明教程

Rushi 是面向中文长音频整理的桌面转写与校对工具。它把录音、转写稿、术语表、纠错记忆和交付导出放在同一个项目工作流里，适合课程录音、访谈、口述整理和需要反复校对的长文本。

这份教程按真实使用顺序编排：先装好应用，再创建项目，接着选择本机或在线转写，最后在 Editor 中听写校对并导出。

## 1. 安装与首次启动

### macOS

下载 macOS DMG 后拖入 Applications。若系统提示应用来自未识别开发者，可以在 Finder 中按住 Control 点击应用，再选择打开。

Rushi 会在首次启动时准备本机 ASR 所需的模型文件。随包默认模型会被复制到应用数据目录，后续可以在断网状态下使用本机转写。

### Windows

Windows 推荐下载“离线安装包 zip”。请先完整解压 zip，再运行同级安装程序。若 SmartScreen 出现拦截提示，确认来源是 Rushi 官网或 GitHub Release 后继续安装。

有 NVIDIA 显卡的 Windows 用户，可以在应用的环境页下载 CUDA 侧车以获得更好的本机转写性能。GPU 组件不是首装必需项，下载失败时仍可使用 CPU 转写。

### 当前平台范围

Rushi 目前面向 macOS 与 Windows 桌面端发布。Linux 桌面端暂不作为正式发布目标。

## 2. 认识三个核心界面

Rushi 的主流程由三个界面组成：

| 界面 | 作用 |
|---|---|
| Welcome | 查看最近项目、创建项目、打开已有项目、搜索项目 |
| Project Hub | 管理项目内的音频、文本、字幕、场次信息和批量任务 |
| Editor | 对单个音频进行转写、播放、波形定位、逐段校对和导出 |

通常你会从 Welcome 新建或打开项目，在 Project Hub 管理文件，再进入 Editor 处理某一条音频。

## 3. 创建项目与导入素材

1. 在 Welcome 中选择新建项目。
2. 填写项目名称，按需要补充讲者、录制时间、地点、主题、整理人等场次信息。
3. 导入本地音频文件。
4. 进入 Project Hub 后，可以继续导入更多音频、文本或字幕文件。

Rushi 会把项目和语段数据保存到本地 SQLite 数据库。音频文件会按项目归入媒体目录，便于后续迁移和备份。

导入时如果发现同一文件已经存在，Rushi 会提示你处理重复导入。导入字幕或已有转录稿时，如果文件名和音频文件名匹配，Rushi 可以将字幕附加到对应音频；如果匹配到多个音频，会让你选择目标文件。

## 4. 准备转写方式

Rushi 支持两类转写来源：本机 ASR 和在线 STT。

### 本机 ASR

本机 ASR 是默认路线。它由桌面应用拉起本机 sidecar 服务，默认地址为 `127.0.0.1:8741`。在环境页的“本机 ASR”中可以查看状态、准备模型、重启运行时或清理模型缓存。

适合选择本机 ASR 的场景：

- 音频内容敏感，不希望上传到云端。
- 网络不稳定，或需要断网工作。
- 希望长期控制成本。

如果状态显示模型未就绪，先在环境页完成模型准备。Windows 用户如配置 CUDA 侧车失败，可以先用 CPU 路线继续工作。

### 在线 STT

在线 STT 是可选路线，适合希望借助云端识别能力、已有厂商额度或需要特定厂商能力的用户。

当前产品中已接入的在线 STT Provider 包括：

- 阿里云百炼语音识别（Fun-ASR）
- 讯飞极速录音转写大模型
- OpenAI
- AssemblyAI
- Deepgram
- 自定义兼容代理

在环境页的“在线 STT”中启用服务、选择厂商、保存密钥并探测连接。讯飞需要 AppID、APIKey、APISecret 三件套；其他厂商通常保存 API Key 即可。在线端点必须使用 HTTPS，本机代理可以使用 localhost 或 `127.0.0.1`。

## 5. 使用术语表提高识别稳定性

在转写前，建议先把人名、地名、佛学名词、机构名、课程名等专有词加入术语表。Rushi 会在支持的转写引擎中把术语作为识别偏置提交。

不同 Provider 的术语使用方式略有差异：

| Provider | 术语偏置方式 |
|---|---|
| 本机 ASR | 随转写请求提交热词 |
| OpenAI | 写入转写提示词 |
| AssemblyAI | 作为 keyterms 提交 |
| Deepgram | 作为 keywords 提交 |
| 阿里云百炼 | 同步为厂商热词表 |
| 讯飞极速 | 作为热词提交 |
| 自定义代理 | 随请求提交 |

术语表只负责“希望识别成什么正形”。如果你在校对时经常把同一个错字改成同一个正确写法，Rushi 会形成纠错记忆，后续可用于规则纠错和智能改稿。

## 6. 自动转写

进入 Editor 后，选择本机 ASR 或已配置好的在线 STT，然后开始自动转写。转写完成后，Rushi 会把语段、时间码、正文和置信信息写入项目数据库。

Project Hub 中也可以对多个文件发起批量转写。批量过程中你可以停止任务；已完成、跳过和失败的数量会在活动记录中显示。

如果当前文件已有手工修改，重新自动转写前会出现确认提示，因为新的转写结果可能覆盖当前语段。建议先保存或导出重要内容。

## 7. 在 Editor 中校对文稿

Editor 是 Rushi 的主工作区。它把波形、播放头、语段区域和文本编辑器绑定在一起，目标是让你一边听，一边快速定位并修正文稿。

### 选择与定位

- 点击文本语段：选中该语段，并跳到语段开头。
- 再次点击已选语段：只移动光标，不重复跳转。
- 点击波形中的语段：选中语段并定位到语段开头。
- 再次点击已选波形语段：按点击位置精确跳转。
- 点击空白波形：移动播放头。
- 使用键盘上下移动：只改变选择，避免打断当前播放定位。

### 波形与播放

波形区支持缩放、适配全局、适配选区、迷你地图和播放跟随。长音频校对时，可以先用迷你地图定位大范围，再放大局部处理细节。

播放支持全局播放、语段播放和循环播放。调整播放速度时，Rushi 会尽量保持人声自然，避免简单变速导致音高明显变化。

### 语段整理

常见校对动作包括：

- 修改正文。
- 拆分过长语段。
- 合并相邻语段。
- 冻结已经确认的语段。
- 查找与替换重复错误。
- 使用规则纠错处理稳定的错词。
- 在定稿模式中做终检和智能改稿。

冻结语段会保留当前分段和正文，全局通读时会跳过，正文也不能直接编辑。需要继续修改时先解冻。

## 8. 保存、关闭与防误操作

Rushi 会在编辑器中显示当前保存状态。面包屑或底部状态出现提示时，说明仍有未保存改动、正在保存或已经保存。

当你准备离开当前文件、关闭项目或退出应用时，如果存在未保存编辑，Rushi 会先弹出确认。转写或批量转写正在进行时，也会阻止直接跳转，避免任务中断或结果丢失。

推荐习惯：

1. 每处理完一段重要内容，先确认保存状态。
2. 重新转写前，先导出或保存当前稿件。
3. 批量转写时，不要直接关闭应用，先停止或等待完成。

## 9. 导入已有转录稿或字幕

如果你已经有 TXT、SRT 或其他文本稿，可以从 Project Hub 或 Editor 的导入入口加入项目。

常见用法：

- 把已有字幕附加到同名音频上。
- 用新的字幕替换当前文件的旧语段。
- 导入纯文本作为项目内独立文本文件。
- 从其他设备带回 Rushi 内容包，再导入恢复项目。

导入字幕到当前 Editor 时，Rushi 会走替换流程；如果当前稿件尚未保存，会先经过关闭保护提示。

## 10. 导出与交付

Rushi 支持普通导出和交付 Word 导出。

### 普通导出

日常整理可以导出：

- TXT：适合继续复制、整理或发给他人。
- SRT：适合字幕、剪辑和视频流程。
- DOCX：适合文档交付。

### 交付 Word

需要正式交付时，可以从 Editor 进入“定稿模式”，完成规则纠错、智能改稿和终检，再打开“交付导出 Word”。

交付 Word 支持三种形态：

| 形态 | 适用场景 |
|---|---|
| 逐字稿 | 保留时间码和逐段正文，适合审校、回听、归档 |
| 讲稿 | 按自然段整理，适合阅读和发布前整理 |
| 干净稿 | 去掉时间码和低置信标记，适合最终阅读稿 |

项目中的讲者、录制时间、地点、主题等信息可以进入 Word 文档抬头。需要时也可以附加修订摘要。勾选大模型润色时，应用会在导出过程中处理明显错别字和错误标点，并以 Word 修订轨呈现可见差异。

### 内容包导出

如果要迁移或备份，可以导出当前项目或整库内容包。整库内容包会包含多个项目和词表信息，适合换电脑、备份或团队内交接。

## 11. 偏好设置

Rushi 会记住一些与校对体验有关的偏好，例如：

- 波形缩放比例。
- 波形高度。
- 转录文本字号。
- 全局播放速度。
- 迷你地图开关。
- 播放跟随方式。
- 文稿播放跟随方式。
- 语段列表筛选。
- 主题和强调色。

这些设置适合根据个人校对习惯慢慢调整。长时间校对时，优先调好字号、波形高度和播放速度，效率提升会很明显。

## 12. 迁移、备份与诊断

### 迁移媒体目录

Rushi 的媒体文件保存在配置的媒体目录下。需要换硬盘或同步到其他位置时，可以调整媒体根目录，并通过项目或内容包导出完成迁移。

为了跨 macOS 和 Windows 使用，Rushi 会尽量保存相对路径。移动项目时，建议使用内容包导出和导入，而不是手工搬散文件。

### 导出诊断包

遇到转写失败、模型准备异常、导出失败或应用状态不明时，可以在应用内导出诊断 zip。诊断包会帮助定位版本、运行时、环境和最近错误。

### 应用内更新

支持应用内更新的版本会从 `updates.rushi.app` 获取更新清单。确认后，应用会下载、验签并安装更新，完成后自动重启。若应用内更新不可用，可以从官网或 GitHub Release 手动下载安装新版。

## 13. 常用快捷键

| 快捷键 | 作用 |
|---|---|
| Cmd/Ctrl + , | 打开设置 |
| Cmd/Ctrl + F | 查找与替换 |
| Cmd/Ctrl + J | 与下一条语段合并 |
| Cmd/Ctrl + K | 与上一条语段合并 |
| Cmd/Ctrl + D | 在播放头位置拆分语段 |
| Space | 播放或暂停 |
| Shift + Space / Ctrl + Space | 正文编辑时播放或暂停 |
| Esc | 关闭面板或对话框 |

不同系统和输入焦点下快捷键可能略有差异；以应用底部提示和菜单为准。

## 14. 常见问题

### 本机 ASR 显示未就绪怎么办？

先到“环境 → 本机 ASR”查看模型、FFmpeg 和 sidecar 状态。若模型未准备完成，等待复制或下载完成；若 sidecar 异常，尝试重启运行时。Windows 上 CUDA 侧车失败时，可以改用 CPU。

### 在线 STT 配好了但不能转写怎么办？

先在“环境 → 在线 STT”探测连接。检查密钥是否保存、网络是否可访问、端点是否为 HTTPS。讯飞需要确认 AppID、APIKey、APISecret 都填写完整。

### 转写后专名还是错怎么办？

把正确写法加入术语表，再重新转写或使用规则纠错。对于已经反复手改的错词，可以把稳定规则纳入纠错记忆。

### 文稿改乱了怎么办？

先不要重新自动转写。确认当前保存状态，必要时先导出 TXT 或 DOCX 留底，再决定是否替换导入、重新转写或从内容包恢复。

### 为什么不能直接离开当前页面？

如果正在转写、批量转写或存在未保存改动，Rushi 会先拦截离开操作。这是为了避免语段丢失、任务中断或新结果覆盖旧稿。

## 15. 推荐工作流

第一次使用可以按这条路线走：

1. 安装 Rushi，并确认本机 ASR 就绪。
2. 新建项目，补充场次信息。
3. 先录入常见专名到术语表。
4. 导入音频并自动转写。
5. 在 Editor 中按波形逐段校对。
6. 对重复错词使用查找替换或规则纠错。
7. 冻结已经确认的语段。
8. 进入定稿模式做终检。
9. 导出逐字稿、讲稿或干净稿。
10. 需要迁移时导出项目或整库内容包。

把“项目、音频、语段、术语、纠错和导出”都留在 Rushi 内完成，后续回看、修改和再次交付会更稳。
